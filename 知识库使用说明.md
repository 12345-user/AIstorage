# 知识库使用说明

这是按“AI + Obsidian 通用知识库结构蓝图”配置的 Vault，并已升级为“项目制 + 公共资料库”结构。

全库统一标准见：`06_Skills/知识库输入标准与指令识别规范.md`。

## 快速入口

- 新资料先放入 `00_Inbox`。
- 自动抓取资料放入 `00_Inbox/自动抓取`。
- 手动剪藏资料放入 `00_Inbox/手动剪藏`。
- 原始资料整理到 `01_Sources`。
- 观点、概念、案例和方法沉淀到 `02_Knowledge`。
- 选题和长期研究放入 `03_Topics`。
- 文章、报告、脚本和课程放入 `04_Outputs`。
- 每日、每周、每月复盘放入 `05_Review`。
- 具体项目资料优先放入 `10_Projects`。
- 多项目复用资料放入 `20_Shared`。

## 最小输入格式

```text
标题：
链接 / 内容：
PROJECT：P1 / P2 / P3 / P4 / P5 / X / 待判断
CMD：整理 / 入库 / 知识卡 / 选题 / 脚本 / 文章 / 报告 / 周报 / 简报
```

## 完整指令格式

```text
CMD:整理 | PROJECT:P4 | ACTION:B+A | OUTPUT:知识卡 | TARGET:02_Knowledge/工具卡片 | PRIORITY:中
```

## 一键全流程指令

### P1-HORIZON-FULLFLOW

用途：处理 P1 客栈 / p1horizon 这类“项目当前进展”资料时，一次性完成整理、入库、知识卡、选题、脚本、文章、报告、周报和每日简报。

推荐写法：

```text
CMD:全流程 | FLOW:P1-HORIZON-FULLFLOW | PROJECT:P1 | ACTION:P+A+S+C | OUTPUT:知识卡+选题+视频脚本+文章+报告+周报+每日简报 | PRIORITY:高

资料来源：
...
```

默认执行结果：

- 整理 / 入库：进入 `10_Projects/01_客栈自媒体脚本与热点/00_项目收件箱`。
- 知识卡：进入 `02_Knowledge` 的案例卡片、方法论卡片或观点卡片。
- 选题：进入 `03_Topics/选题池`。
- 脚本：进入 `04_Outputs/视频脚本`；只有进入拍摄或发布时才同步到 P1 项目 `04_脚本成品`。
- 文章：进入 `04_Outputs/文章`。
- 报告：进入 `04_Outputs/报告`。
- 周报：进入 `04_Outputs/周报总结`。
- 简报：进入 `05_Review/每日简报`。
- 复盘：只有产生发布数据、决策变化或明确经验时，才进入 P1 项目 `05_发布复盘`。

适用资料：

- 客栈网站项目进展。
- 客栈素材资产盘点。
- 客栈上线前资料清单。
- 客栈自媒体阶段性复盘。
- p1horizon 相关代码、素材和内容运营资料。

## 推荐流程

1. 收集信息到 `00_Inbox`。
2. 用统一标准补齐“标准识别区”。
3. 先判断资料属于 P1-P5 哪个项目，或是否属于共享资料 X。
4. 按相关性、新鲜度、价值度、可输出性、关联性做判断。
5. 将资料分流到 `10_Projects`、`20_Shared`、`01_Sources`、`02_Knowledge` 或 `03_Topics`。
6. 定期从项目资料、主题和知识卡片生成 `04_Outputs`。
7. 在 `05_Review` 中记录复盘和知识库优化动作。

## 文档重要性与去重规则

整理文档时先判断重要性，再决定入库、合并或删除。

### 高重要性：必须保留

- 原始资料中含有后续无法稳定复原的信息，例如真实数据、路径、链接、截图说明、代码检查结果、会议记录、实验结果。
- 直接支持项目决策或行动，例如上线清单、修复任务、课程截止日期、发布计划。
- 已经形成可交付成果，例如报告、文章、脚本、周报、实验报告。
- 能长期复用的方法、案例、观点或工具知识卡。

### 中重要性：合并或归档

- 阶段性总结、每日/每周复盘、资料清单、待办列表。
- 与已有笔记部分重复，但包含少量新增判断或下一步动作。
- 对单个项目有用，但不适合沉淀为全库知识卡。

处理方式：优先合并到项目页、报告、周报或清单；合并后原文件可删除。

### 低重要性：默认删除或不入库

- 只重复另一篇笔记内容的同步副本、索引副本、空壳复盘。
- 没有新增事实、判断、行动或链接的流水账。
- 已被更新版完全替代的旧稿。
- 可以随时从高质量源文件重新生成的临时草稿。

### 去重顺序

1. 保留原始证据或最完整源文件。
2. 保留最终交付件。
3. 保留一张最清晰的知识卡或方法卡。
4. 删除只做跳转、同步或重复摘要的副本。
5. 如果两篇内容各有价值，合并后删除较弱的一篇。

## 项目代码

- P1：客栈自媒体脚本与热点。
- P2：生物信息研究生课程与实验报告。
- P3：大学课程日常跟进。
- P4：AI 前沿与 Codex 工具。
- P5：其他杂事项目。
- X：共享资料。

## 分类动作

- `P`：进项目。
- `X`：进共享区。
- `S`：进选题池。
- `A`：做知识卡。
- `B`：存参考资料。
- `C`：归档。
- `D`：忽略。

## 删除判定

允许删除的文件必须同时满足：

- 已确认有更完整或更高质量版本存在。
- 删除后不会丢失原始事实、关键链接、任务清单或交付成果。
- 文件状态属于低重要性，或已被合并。

不建议删除：

- 原始资料。
- 已发布输出。
- 含测试结果、项目决策、真实配置要求的报告。
- 用户明确要求保留的笔记。

- 可用的 API Key 或账号授权。
- p1horizon 后续需补：真实客栈名称、定位、地址、联系方式、预订入口、房型价格、入住须知、真实活动和主理人资料。
