## 一句话摘要

这是评估一个信息收集系统是否完整、可维护、可复用的 checklist，适用于 AI 信息收集、客栈热点、文献追踪、课程资料、竞品监控等场景。

## MVP 检查

- [ ] 是否定义了项目目标和目标用户？
- [ ] 是否定义了高价值内容标准？
- [ ] 是否至少有 3 个稳定信息源？
- [ ] 是否每个信息源可以独立启用/禁用？
- [ ] 是否有统一内容模型？
- [ ] 是否保存原始抓取结果？
- [ ] 是否有 URL 去重？
- [ ] 是否有基础评分或筛选规则？
- [ ] 是否能生成 Markdown 或 JSON 输出？
- [ ] 是否有本地归档目录？
- [ ] 是否有运行日志？
- [ ] 是否确保密钥不入库？

## 完整生产检查

- [ ] 是否支持多来源并发采集？
- [ ] 是否单源失败不影响全局？
- [ ] 是否支持来源特有 metadata？
- [ ] 是否支持标题相似度或主题级去重？
- [ ] 是否支持 AI 评分和摘要？
- [ ] 是否支持评分阈值配置？
- [ ] 是否支持高分内容背景补全？
- [ ] 是否支持多格式输出？
- [ ] 是否支持多渠道分发？
- [ ] 是否支持定时运行？
- [ ] 是否保存中间阶段产物？
- [ ] 是否记录 token / cost / API 错误？
- [ ] 是否支持从中间阶段重跑？
- [ ] 是否有人工复核入口？
- [ ] 是否有测试覆盖？
- [ ] 是否提供 MCP 或 API 给其他工具调用？

## 安全检查

- [ ] `.env` 是否加入 `.gitignore`？
- [ ] config 中是否只保存环境变量名？
- [ ] 是否没有真实 API Key、token、cookie、Webhook URL？
- [ ] GitHub Token 是否最小权限？
- [ ] 私有 RSS token 是否通过环境变量注入？
- [ ] 是否有密钥泄露后的轮换流程？

## 鲁棒性检查

- [ ] 采集失败是否记录错误并继续？
- [ ] AI JSON 解析失败是否有默认回退？
- [ ] 背景补全失败是否保留原始摘要？
- [ ] 推送失败是否保留本地输出？
- [ ] 是否有重试和节流？
- [ ] 是否限制最大抓取数量？
- [ ] 是否限制最大正文长度？
- [ ] 是否限制并发数？

## 特异性检查

- [ ] 是否写明领域关注重点？
- [ ] 是否写明排除内容？
- [ ] 是否写明人工确认内容？
- [ ] 是否写明输出风格？
- [ ] 是否写明目标项目归属？
- [ ] 是否写明对应 Obsidian 保存位置？

## 评分参考

| 分数 | 状态 | 判断 |
| --- | --- | --- |
| 0-5 | 未成型 | 只有零散抓取或手动整理 |
| 6-10 | MVP | 能采集、去重、输出，但不够稳定 |
| 11-18 | 可用 | 有评分、归档、日志和基本失败回退 |
| 19-25 | 准生产 | 支持多源、补全、分发和成本控制 |
| 26+ | 生产级 | 有 MCP/API、测试、观测、人工复核和安全闭环 |
